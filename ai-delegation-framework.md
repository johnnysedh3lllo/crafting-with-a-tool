# AI Delegation Framework

## Principles

1. Understand anything proportional to how much it affects the system's behaviour. This governs ownership.
2. Implement anything proportional to how much it affects my improvement. This governs competence.

---

## Tier 1

Product decisions and architectural decisions.

- What should this feel like?
- How should data flow?
- What are the tradeoffs between approaches?

If something falls under these kinds of questions, I must be fully involved in them.

---

## Tier 2

What goes wrong if this is done badly?

If the answer is significant, I engage with it and internalise it well enough to modify and work on it myself.

If not, consider delegation but review to ensure quality is maintained.

---

## Tier 3

- Do I know how to do this already?
- Is doing it myself friction against the actual goal?

If so, consider delegation.

---

## Skill Formation

To decide what should be a priority for skill formation, it is important to ask what the goal is:

- Why am I doing this?
- Why should it be by hand?
- Why is it necessary that I delegate?

The goal defines the mode of approach.

**If the goal is learning or practice** — implementing by hand is the point. A router built from scratch to understand routers needs to be built by hand.

**If the goal is immediate delivery** — delegation might be a better option, given that the environment may be against the sought learning. A migration script in production, in service of a deadline, is one example.

Beyond these, the real question is confidence. If I'm already confident I could do the thing, AI is just there to speed it up — delegate. If I'm not, or I want to actually know I can rather than assume it, then I might need evidence as to if I can or cannot, and that's a reason to implement it by hand.

---

## If You Digress

Ask: what tier is this?

- If Tier 1 — do it.
- If Tier 2 — engage with it and internalise it well enough to modify and work on it.
- If Tier 3 — delegate.
